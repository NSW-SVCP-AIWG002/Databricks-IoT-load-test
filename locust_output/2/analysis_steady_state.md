> 定常状態 RPS が理論値とほぼ完全に一致しており、全 20,500台が正常に 90秒間隔で送信できていることを確認。

### エンドツーエンド パイプライン遅延

パイプライン：デバイス送信（`event_timestamp`）→ IoT Hub → Event Hub → Databricks Streaming → silver_sensor_data 書き込み（`create_time`）

| 指標 | MySQL | Unity Catalog（Databricks） |
|---|---|---|
| 平均遅延（avg_lag_sec） | 583.16 秒（約 **9.7 分**） | 402.89 秒（約 **6.7 分**） |
| 最大遅延（max_lag_sec） | 959.09 秒（約 **16.0 分**） | 726.00 秒（約 **12.1 分**） |

> MySQL と Databricks の値が異なる理由：MySQL の `UNIX_TIMESTAMP()` はセッションタイムゾーン依存のため変換結果がずれる可能性がある。Unity Catalog 側の値がより信頼性が高い。

**考察：**

- 対象期間を定常状態に絞っても平均遅延 402秒（約6.7分）が観測されている。
- これは定常状態においても Event Hub バックログ（ramp-up 中に蓄積した未処理メッセージ）の解消処理が継続しているためと推定される。
- バックログが完全に解消した後の純粋な定常遅延を計測するには、さらに後半の時間帯（ramp-up 完了から十分な時間が経過した後）を対象にする必要がある。

### Event Hub Namespace（ehns-nsw-iotbricks-dev）

#### Throttled Requests / Incoming Messages

![Event Hub - Throttled Requests と Incoming Messages](./ehns1.png)

| メトリクス | 合計値 |
|---|---|
| Throttled Requests（合計） | **192.49k** |
| Incoming Messages（合計） | **136.67k** |
| Throttle 率（Throttled / Incoming） | **約 141%** ※ |

> ※ Throttled Requests が Incoming Messages を上回っているのは、再試行リクエストも計上されるため。

**考察：**
- **Throttled Requests 192.49k は深刻なボトルネック**。Incoming Messages（136.67k）に対してスロットリングが大量発生しており、これがパイプライン遅延（平均 402秒）の主要因と推定される。
- グラフでは 02:25 頃に Throttled Requests が約 37k のピークを記録しており、この時点でバックログ解消処理が集中していたと考えられる。

**Throttled Requests 大量発生の原因：**
- **負荷テスト固有の高スループット**：本番想定の送信間隔は 10分間隔（約 34.2 req/s）だが、今回の負荷テストは 90秒間隔（約 227.8 req/s）で実施しており、**本番の約 6.7倍の負荷**をかけている。この高スループットが Event Hub のスループットユニット（TU）上限を超えたことでスロットリングが発生した。
- **本番環境（10分間隔）では同様のスロットリングは発生しない見込み**。現在の TU 構成は本番想定負荷（34.2 req/s）に対しては十分な余裕がある。

#### メッセージフロー詳細

![Event Hub - メッセージフロー](./ehns2.png)

| メトリクス | 値（02:29 時点） |
|---|---|
| Incoming Messages（合計） | **13.79k** |
| Outgoing Messages（合計） | **72.55k** |
| Captured Messages（合計） | **13.64k** |
| Capture Backlog（平均） | **24.62k** |

**考察：**
- Outgoing Messages（72.55k）が Incoming（13.79k）の約 **5.3倍**：Databricks が ramp-up 中のバックログを一括で読み出している。
- Capture Backlog 平均 24.62k：定常状態の 10分間でもバックログが残存しており、完全解消には至っていない。
- Incoming と Captured がほぼ一致（13.79k vs 13.64k）：新規メッセージはほぼ漏れなくキャプチャされている。
